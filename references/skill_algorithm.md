# Requirements Extraction v2

## Когда использовать этот навык

Применять всякий раз, когда пользователь передаёт неструктурированный текст (письмо заказчика, протокол встречи, интервью, описание «бизнес-потребностей», концепция продукта) и просит:

- извлечь / выделить требования
- сформировать ТЗ / спецификацию
- разобрать бизнес-потребности
- проверить полноту требований

Работать строго на языке входных данных. Если текст на русском — вся выдача на русском. Если на английском — на английском.

## Методологическая база

| Методология | Роль в конвейере |
|---|---|
| **BABOK v3** | Эталон качества, верификации и валидации требований (атрибуты: корректность, полнота, непротиворечивость, осуществимость, необходимость, приоритетность, однозначность, проверяемость) |
| **EARS** (Easy Approach to Requirements Syntax) | Нормализация текста в атомарные утверждения. Шаблоны: Ubiquitous / Event-driven / State-driven / Unwanted behaviour / Optional feature |
| **Volere ReqSpec v16** | Структура документа, типы требований, атрибуты «снежной карточки» (snow card), fit criterion, трассируемость |
| **Volere JSON Schema** | Финальный формат вывода; см. `references/volere_schema_summary.md` |

## Конвейер обработки (6 этапов)

### Этап 1 — Извлечение кандидатов

**Цель:** разбить входной текст на смысловые фрагменты, каждый из которых потенциально несёт одно утверждение о системе/продукте.

**Действия:**
1. Прочитать весь входной текст целиком.
2. Выделить предложения / фрагменты, описывающие: цели, проблемы, ожидания, ограничения, правила, качества, роли, данные, события.
3. Пронумеровать каждый фрагмент-кандидат: `[SRC-N]` с цитатой (дословно) или ссылкой на раздел.
4. Фрагменты, не являющиеся требованиями, но важные по контексту, маркировать: `[CONTEXT]`, `[ASSUMPTION]`, `[RISK]`, `[OPEN_ISSUE]`, `[POLICY]`.
5. **Не изобретать требований.** Если смысл фрагмента неоднозначен — зафиксировать неопределённость и занести в список уточняющих вопросов (этап 5).

**Формат промежуточного вывода (внутренний — показывать пользователю по запросу):**
```
[SRC-1] «<цитата>» → кандидат
[SRC-2] «<цитата>» → [ASSUMPTION]
```

### Этап 2 — Классификация по типам

**Применять классификатор по Volere ReqSpec v16:**

| Код типа | Название | Описание |
|---|---|---|
| `BR` | Business Requirement | Цель или результат, нужный бизнесу; «зачем» |
| `UR` | User Requirement | Что пользователь должен уметь делать с продуктом |
| `FR` | Functional Requirement | Что система должна делать (поведение, функции) |
| `NFR-PERF` | Performance | Быстродействие, пропускная способность, латентность |
| `NFR-USAB` | Usability | Лёгкость использования, обучение, доступность |
| `NFR-LOOK` | Look & Feel | Внешний вид, фирменный стиль |
| `NFR-SECU` | Security | Аутентификация, авторизация, шифрование, аудит |
| `NFR-OPER` | Operational & Environmental | Окружение, инфраструктура, режим работы |
| `NFR-MAIN` | Maintainability & Support | Сопровождение, логирование, диагностика |
| `NFR-CULT` | Cultural | Локализация, язык, нормы |
| `NFR-LEGL` | Legal | Законодательство, стандарты, лицензии |
| `CON` | Constraint | Ограничение: бюджет, сроки, платформа, технология |
| `ARCH` | Architectural Constraint | Архитектурное или дизайн-ограничение |
| `ASS` | Assumption | Допущение, принятое аналитиком |
| `OI` | Open Issue | Открытый вопрос, требующий уточнения |
| `RISK` | Risk | Идентифицированный риск |

**Правило для сложных случаев:** если кандидат не укладывается однозначно в один тип, зафиксировать оба варианта с пояснением неопределённости и вынести уточняющий вопрос. Не упрощать принудительно.

---

### Этап 3 — Нормализация в атомарные требования по EARS/Volere

**Принцип атомарности:** одно требование — одно поведение/свойство. Если кандидат содержит «и» — проверить, не два ли это разных требования.

**EARS-шаблоны (применять по ситуации):**

| Шаблон | Синтаксис | Пример |
|---|---|---|
| Ubiquitous | `Система должна <действие>` | «Система должна вести журнал всех операций» |
| Event-driven | `Когда <событие>, система должна <действие>` | «Когда пользователь нажимает Сохранить, система должна валидировать форму» |
| State-driven | `Пока <состояние>, система должна <действие>` | «Пока сессия активна, система должна обновлять токен каждые 15 минут» |
| Unwanted behaviour | `Если <нежелательное условие>, система должна <защитное действие>` | «Если платёж отклонён, система должна уведомить пользователя и предложить повтор» |
| Optional feature | `Если продукт включает <функцию>, то система должна <действие>` | «Если продукт включает модуль отчётности, система должна поддерживать экспорт в PDF» |

**Атрибуты Volere snow card для каждого требования:**

```
req_id          — уникальный идентификатор (BR-001, FR-007 и т.д.)
req_type        — тип из классификатора выше
description     — текст по EARS-шаблону
rationale       — обоснование: почему это нужно
source          — цитата из входных данных или [SRC-N]
originator      — стейкхолдер, выдвинувший требование (если известен)
fit_criterion   — измеримый критерий приёмки (ОБЯЗАТЕЛЕН для FR, NFR)
priority        — high / medium / low / tbd
conflicts       — список req_id конфликтующих требований
status          — extracted / clarification_needed / assumed
notes           — пояснения, допущения, неопределённости
```

**Правило fit_criterion:** если критерий нельзя сформулировать измеримо — требование либо расплывчатое (вынести OI), либо является бизнес-контекстом (маркировать `[CONTEXT]`).

### Этап 4 — Проверка качества по BABOK

**Применить 8 атрибутов качества требований (BABOK v3, §6.3.2):**

| Атрибут | Проверка |
|---|---|
| **Корректность** | Требование точно отражает намерение стейкхолдера |
| **Полнота** | Все аспекты сценария описаны; нет «дыр» |
| **Непротиворечивость** | Нет конфликтов внутри набора требований |
| **Осуществимость** | Требование реалистично при заданных ограничениях |
| **Необходимость** | Требование действительно нужно (не «хотелка») |
| **Приоритетность** | Всем требованиям назначен приоритет или пояснено, почему нет |
| **Однозначность** | Текст допускает только одну интерпретацию |
| **Проверяемость** | Наличие fit_criterion, позволяющего проверить выполнение |

**Результаты этапа:**
- Список выявленных нарушений с req_id и описанием проблемы.
- Список конфликтов / противоречий между требованиями.
- Пробелы в покрытии (gap analysis): что упомянуто в бизнес-контексте, но не перешло в требования.

---

### Этап 5 — Уточняющие вопросы

**Правила:**
- Каждый вопрос должен быть конкретным, а не риторическим.
- Каждый вопрос имеет ответ, который меняет содержание требования или устраняет неопределённость.
- Ссылаться на req_id или [SRC-N], к которому относится вопрос.
- Ранжировать по приоритету: `[BLOCKER]` (блокирует классификацию), `[HIGH]` (влияет на fit_criterion), `[MEDIUM]` (детализация), `[LOW]` (предпочтения).

**Формат:**
```
Q-001 [BLOCKER] → req_id/SRC-N: <текст вопроса>
  Почему важно: <что изменится в требовании в зависимости от ответа>
  Варианты ответа: A) ... B) ... (если применимо)
```

**Задавать вопросы немедленно** — не копить до конца обработки. Если вопрос возник на этапе 1 или 2, задать его сразу, затем продолжить с допущением, которое явно маркируется как `[ASSUMPTION]`.

---

### Этап 6 — Финальный вывод

#### 6a. JSON-файл (вложение)

Сформировать JSON, строго соответствующий схеме `volere_requirements_schema.json` (см. `references/volere_schema_summary.md`).

**Обязательные разделы:**
- `document_meta` — метаданные документа
- `project_drivers` — цели, стейкхолдеры (заполнять по мере доступности; недостающее → `"tbd"` или `null`)
- `project_constraints` — ограничения
- `functional_requirements` — функциональные требования
- `non_functional_requirements` — нефункциональные требования
- `project_issues` — открытые вопросы, риски, допущения

Сохранить файл как `requirements_<project_name>.json` и передать пользователю через `share_file`.

#### 6b. Сводная таблица

Вывести Markdown-таблицу:

| ID | Тип | EARS-формулировка | Источник | Fit Criterion | Приоритет | Конфликты | Статус |
|---|---|---|---|---|---|---|---|
| BR-001 | BR | ... | [SRC-1] | — | High | — | extracted |

#### 6c. Краткое резюме

- Количество извлечённых требований по типам.
- Количество открытых вопросов.
- Выявленные конфликты.
- Пробелы в покрытии.
- Рекомендации по следующим шагам.

## Критические правила

1. **Не изобретать требований.** Извлекать только явное или косвенно подразумеваемое из входных данных.
2. **Маркировать допущения.** Каждый `[ASSUMPTION]` указывается явно с обоснованием.
3. **Сохранять трассируемость.** Поле `source` в каждом требовании — цитата или [SRC-N].
4. **Язык входных данных = язык вывода.** Не переключаться на английский, если входной текст на русском.
5. **Не упрощать сложные случаи.** Неопределённость → OI + вопрос, а не произвольный выбор.
6. **Fit criterion обязателен** для всех FR и NFR. Если невозможно сформулировать — требование расплывчатое.
7. **Задавать уточняющие вопросы по мере появления**, не в конце.

## Формат промежуточного представления агенту

При обработке больших текстов структурировать внутреннее состояние:

```
STATE = {
  candidates: [{id, quote, tag}],          # этап 1
  classified: [{id, type, explanation}],   # этап 2
  normalized: [{snow_card_fields...}],     # этап 3
  quality_issues: [{req_id, issue, attr}], # этап 4
  questions: [{id, priority, req_id, text, why, options}], # этап 5
  output_json: {...},                      # этап 6a
  output_table: [...],                     # этап 6b
}
```

## Быстрый старт для агента

1. Получить входной текст.
2. Определить язык входных данных → установить язык вывода.
3. Если нет информации о проекте (название, заказчик) → задать `Q-000 [BLOCKER]` сразу.
4. Запустить этапы 1–5 последовательно.
5. После этапа 5 задать все накопленные вопросы пользователю (если не были заданы ранее).
6. Дождаться ответов на `[BLOCKER]` вопросы, затем завершить этап 6.
7. Отдать JSON-файл + таблицу + резюме.
