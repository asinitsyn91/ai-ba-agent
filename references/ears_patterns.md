# EARS — шаблоны нормализации требований

**Easy Approach to Requirements Syntax** (Mavin et al., 2009)

## Пять шаблонов

### 1. Ubiquitous (Всеобщее)
Требование, которое справедливо всегда, независимо от состояния или события.

**Синтаксис:** `Система должна <глагол> <объект>.`

**Примеры:**
- «Система должна сохранять все транзакции в неизменяемом журнале аудита.»
- «Система должна поддерживать русский и английский языки интерфейса.»

---

### 2. Event-driven (Управляемое событием)
Требование, срабатывающее в ответ на конкретное внешнее или внутреннее событие.

**Синтаксис:** `Когда <триггер>, система должна <отклик>.`

**Примеры:**
- «Когда пользователь завершает ввод платёжных реквизитов, система должна валидировать формат карточного номера по алгоритму Луна.»
- «Когда баланс счёта опускается ниже порогового значения, система должна отправить push-уведомление владельцу счёта.»

---

### 3. State-driven (Управляемое состоянием)
Требование, действующее непрерывно, пока система находится в определённом состоянии.

**Синтаксис:** `Пока <состояние системы или пользователя>, система должна <поведение>.`

**Примеры:**
- «Пока пользователь не прошёл двухфакторную аутентификацию, система должна ограничивать доступ только к публичным разделам.»
- «Пока очередь задач содержит элементы с высоким приоритетом, система должна обрабатывать их первыми.»

---

### 4. Unwanted Behaviour (Нежелательное поведение)
Требование, описывающее защитную реакцию системы на сбой, ошибку или нежелательное условие.

**Синтаксис:** `Если <нежелательное условие / сбой>, система должна <защитное действие>.`

**Примеры:**
- «Если соединение с базой данных прерывается на этапе записи, система должна откатить транзакцию и вернуть код ошибки 503.»
- «Если пользователь вводит неверный пароль пять раз подряд, система должна заблокировать учётную запись и уведомить администратора.»

---

### 5. Optional Feature (Необязательная функция)
Требование, применимое только если продукт включает определённую опциональную функцию или конфигурацию.

**Синтаксис:** `Если продукт включает <функцию/модуль>, то система должна <поведение>.`

**Примеры:**
- «Если продукт включает модуль аналитики, система должна обеспечивать выгрузку отчётов в форматах XLSX и PDF.»
- «Если в конфигурации включён многоарендный режим, система должна изолировать данные каждого арендатора на уровне схемы БД.»

---

## Критерии выбора шаблона

| Ситуация | Шаблон |
|---|---|
| Требование всегда истинно | Ubiquitous |
| Требование срабатывает по событию | Event-driven |
| Требование действует в период/состояние | State-driven |
| Требование описывает реакцию на ошибку/сбой | Unwanted Behaviour |
| Требование применимо только в определённой конфигурации | Optional Feature |

## Типичные ошибки при нормализации

| Ошибка | Пример неверно | Исправление |
|---|---|---|
| Составное требование | «Система должна валидировать форму и сохранять данные» | Разбить на FR-001 (валидация) + FR-002 (сохранение) |
| Расплывчатый глагол | «Система должна обрабатывать быстро» | «Система должна обрабатывать запрос за не более 2 секунд» |
| Решение вместо требования | «Система должна использовать Redis для кэширования» | `[ARCH]` ограничение или NFR-PERF с fit_criterion |
| Пассивный залог без субъекта | «Данные должны шифроваться» | «Система должна шифровать данные пользователей по AES-256» |
| Отсутствие fit_criterion | «Система должна быть удобной» | Добавить: fit_criterion = «SUS Score ≥ 75 по результатам тестирования с 5 пользователями» |
